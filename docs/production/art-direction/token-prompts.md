# Token Prompts

This file contains final image prompts for the printable token sheets used across the project.

Each prompt assumes the image model is given the matching token-sheet wireframe as the layout reference, plus one or more hero cards as style-reference attachments where noted.

## Page 1: Move, Attack, And Special Planning Tokens

```text
Create a flat, print-ready fantasy board game token sheet that follows the attached diamond-token wireframe exactly.

This is a premium tabletop component for a Dota-inspired board game. The result must look like it belongs to the same manufactured product family as the project's existing cards and printed components, while avoiding direct use of copyrighted Dota 2 UI, logos, or official Valve art.

ATTACHMENTS PROVIDED
- Attachment 1: diamond-token wireframe layout
- Attachment 2: Rubick hero card for style and theme reference only
- Attachment 3: Invoker hero card for style and theme reference only

Use the Rubick and Invoker hero cards only as general design-context references for ornament, palette discipline, material finish, icon rendering quality, and overall fantasy theme cohesion. Do not copy character portraits, named spell icons, or hero-specific card compositions directly into the token sheet.

PRIMARY GOAL
Generate a clean printable diamond-token sheet for these planning-token families only:
1. Move tokens
2. Attack tokens
3. Special spell-access planning tokens for Rubick and Invoker

LOCKED LAYOUT REQUIREMENTS
- Preserve the attached page layout exactly.
- Keep the same page proportions, same rounded outer border, and the same arrangement of rotated rounded-square token cut zones.
- Keep the token cells aligned exactly to the wireframe grid.
- Do not redesign the page architecture.
- Do not change the number, spacing, tilt, or proportions of the token slots.
- Present the page straight-on, perfectly flat, with no perspective distortion.

PRINTABILITY REQUIREMENTS
- This must read as a real printable token sheet, not a mockup.
- Use a bright clean background suitable for printing.
- Keep all token art fully contained inside its own token boundary.
- Leave clean gutters between token cells for cutting or punchboard separation.
- Use crisp outlines and high-contrast iconography.
- Avoid muddy shadows, fog, overpainting, or low-contrast effects that would reduce print legibility.
- Do not place hands, table surfaces, props, scissors, dice, cards, rulers, or environmental scene dressing around the page.
- Do not render the page as a photographed object, angled sheet, curled poster, or 3D product mockup.
- No dramatic cast shadows outside the token artwork.
- No heavy paper texture that interferes with readability.

STYLE DIRECTION
- Match the project's premium fantasy board game visual language and use the attached Rubick and Invoker hero cards as broad styling context for general design and theme.
- Borrow overall product-family qualities such as polished component design, restrained ornament, elegant gold-edged detailing, controlled fantasy palette, material finish, and icon rendering quality.
- The page should feel like a premium manufactured planning-token sheet, not like character art.
- Use a coherent shared frame language across every token.
- Let the iconography and color coding distinguish token families inside that shared system.
- The tokens should stay readable at arm's length on a real tabletop.

TOKEN FRAME SYSTEM
- Every token should use the same manufactured diamond-frame language.
- Each token should have a consistent border treatment with a thin premium rim, subtle inner panel, and a bold central icon area.
- Keep token faces clean and icon-forward.
- Use minimal text.
- Any text or numerals must be large, crisp, and secondary to the icon.

PAGE CONTENT AND REQUIRED COUNTS
Include only these tokens on this page:
- Move 1 Hex x20
- Move 3 Hexes x10
- Move 5 Hexes x10
- Attack x1 x40
- Rubick Stolen Spell 1 x1
- Rubick Stolen Spell 2 x1
- Invoker Invoked Spell 1 x1
- Invoker Invoked Spell 2 x1

Do NOT include item slot tokens.
Do NOT include generic ability tokens.

VISUAL RULES BY TOKEN TYPE
- Move 1 Hex: mobility icon with a single-step arrow or one-segment path marker.
- Move 3 Hexes: same family as Move 1 Hex, but clearly upgraded through three-step motion language or a large numeral 3.
- Move 5 Hexes: same family as the other Move tokens, but clearly the long-distance version with a large numeral 5 or extended travel motif.
- Attack x1: aggressive combat iconography, such as crossed blades, weapon impact, or a clean strike glyph; this should feel distinct from movement at a glance.
- Rubick Stolen Spell 1: arcane theft / lifted magic / spell-copy iconography with a clearly labeled or symbolized slot 1 identity.
- Rubick Stolen Spell 2: same family as Rubick Stolen Spell 1, but clearly marked as slot 2.
- Invoker Invoked Spell 1: invoked-magic / prepared-spell iconography with a clearly labeled or symbolized slot 1 identity.
- Invoker Invoked Spell 2: same family as Invoker Invoked Spell 1, but clearly marked as slot 2.

FAMILY COLOR LANGUAGE
- Move tokens: cool mobility palette, such as teal, blue, or wind-touched jade.
- Attack tokens: warmer offensive palette, such as crimson, ember, bronze, or gold-accented red.
- Rubick special tokens: arcane green-teal stealing-magic identity.
- Invoker special tokens: sun-rune / arcane gold / amber / ivory spell-preparation identity.

REPETITION RULE
- Place the required counts first.
- If the attached wireframe contains more token slots than the required counts, fill the remaining slots only with extra copies of the most-used generic tokens on this page.
- Use this spare-fill priority: Attack x1 first, then Move 1 Hex, then Move 3 Hexes.
- Do not create extra copies of Rubick or Invoker special tokens unless space still remains after the generic-token spare fill.

ICONOGRAPHY RULES
- Use bold central symbols with clean silhouettes.
- Favor icon clarity over painterly detail.
- Use subtle magical embellishment, not noisy illustrative clutter.
- Keep token identity understandable even if printed small.
- No token should depend on tiny text to be recognized.

ANTI-DRIFT / NEGATIVE CONSTRAINTS
- Do not redesign the page layout.
- Do not change the token slot arrangement.
- Do not turn the sheet into cards, stickers, coins photographed on a table, or a punchboard mockup seen at an angle.
- Do not include character portraits or full hero art.
- Do not directly copy Rubick or Invoker spell symbols, card layouts, or hero-specific illustrated elements from the attached reference cards.
- Do not use copyrighted Dota 2 UI symbols or official Valve art.
- Do not make the sheet muddy, hyper-rendered, or overloaded with effects.
- Do not place giant text labels inside each token.

FINAL OUTPUT TARGET
The final image should look like a production-quality printable diamond-token sheet for a premium fantasy board game: clean, aligned, readable, stylistically cohesive, and immediately usable as a print reference for move, attack, and special spell-access planning tokens.
```
